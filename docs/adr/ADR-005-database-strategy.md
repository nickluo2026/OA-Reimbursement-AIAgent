# ADR-005：数据库选型策略（SQLite → MySQL）

- 状态：已采纳
- 日期：2025-07（V1.4 更新）
- 关联：constitution.md §3.1；design.md §1.3、§7.1、§7.1.1
- 作者：架构评审（EA）

## 背景

设计目标为 MySQL 8.0（事务可靠、支撑 §7.1 主从架构），但 V0.x 阶段需聚焦 AI 核心能力验证，
尽快跑通 OCR → 异常检测 → 分类限额主链路，不希望被数据库运维拖累。

## 方案

- **A. 直接上 MySQL 8.0**：符合生产目标，但需提前搭建实例与主从，拖慢初期验证节奏。
- **B. V0.x 用 SQLite 快速验证，V1.0 再迁移 MySQL**：零配置、无需运维，加速迭代；
  迁移时仅替换连接字符串 + 数据迁移脚本。
- **C. 直接引入云数据库**：开箱即用但增加早期成本与依赖。

## 决策

选择 **方案 B**。理由：

1. SQLite 零配置、无需运维，加速 V0.x 能力验证；
2. 业务代码基于 SQLAlchemy ORM 实现，迁移时仅需替换连接字符串 + 数据迁移脚本，业务代码无需修改；
3. 生产环境（目标 V2.0 生产化）再迁移至 MySQL 8.0 以支撑并发与主从架构。

## 影响

- V0.x/V1.4 阶段 `database.py` 使用 SQLite（聚焦 AI 核心能力与 LangGraph 编排验证），截至 V1.4 仍在使用，
  MySQL 迁移顺延至 V2.0；
- 风险：SQLite 不支撑并发写入、无连接池，不可用于生产环境 → 已列管于 §7.1.1 迁移计划
  （V2.0 切换 MySQL 主从）。
