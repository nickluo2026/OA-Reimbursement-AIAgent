# ADR-007：引入 LangGraph 作为智能体编排平台

- 状态：已采纳
- 日期：2026-07（V1.4 已实现）
- 关联：constitution.md §2.4；design.md §2.2、§8.5、§16；ADR-002、ADR-008
- 作者：架构评审（EA）

## 背景

当前 `skill/agent.py` 的 `run_reimbursement_skill()` 将「OCR → 异常检测 → 分类限额」以硬编码线性串联方式实现，
新增 Agent（行程单、火车票、机票）必须修改核心代码，违反开闭原则；无并行编排、失败重试/补偿、
状态持久化与断点续跑能力，难以支撑 V2.0 多票据并行与 V3.0 多 Agent 协作审批建议。

## 方案

| 选项 | 定位 | 与现有架构契合度 | 多 Agent 扩展 | 可视化 | 开源协议 |
|------|------|------------------|--------------|--------|---------|
| **LangGraph** | LangChain 出品，图式工作流编排 | **高**（Function Call → Graph 节点直接映射） | 节点+边定义协作 | 代码定义 | MIT |
| Dify | 可视化低代码 LLM 应用平台 | 中（需封装自定义工具） | 拖拽节点 | **强** | Apache 2.0 |
| AutoGen | Microsoft 出品，多 Agent 对话协作 | 中（需重构为对话模式） | 角色定义 | 弱 | MIT |
| CrewAI | 角色任务协作框架 | 中 | 角色编排 | 弱 | MIT |
| 自研编排 | 自行实现状态机 | 低 | 需重复造轮子 | 无 | — |

## 决策

选择 **LangGraph**，理由：

1. **契合度最高**：现有三功能可直接映射为 `StateGraph` 节点，迁移成本最低，功能保持不变；
2. **状态机原生支持**：条件分支（异常检测拦截则提前返回）、并行节点、循环、Human-in-the-loop 均开箱即用；
3. **断点续跑 & 审计**：内置 checkpointer 支持状态持久化，满足企业级合规追溯；
4. **生态成熟**：LangChain 生态，与 DeepSeek SDK、Function Call 兼容；
5. **开源 MIT**：无商用风险。

## 影响

- 新增 `skill/orchestrator/` 编排层，`agent.py` 的编排逻辑迁移至 `graph.py`（V1.4 已完成重构）；
- 引入 `langgraph` 依赖；
- V2.0 起多 Agent 并行编排；
- 风险与缓解：
  - LangGraph 版本迭代较快 → 锁定主版本，跟随官方 LTS；
  - 团队学习成本 → 编排层独立，提供节点开发规范文档（design.md §16）。
