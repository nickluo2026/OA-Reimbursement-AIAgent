# ADR-003：三态校验模型（通过 / 预警 / 拦截）

- 状态：已采纳
- 日期：2025-07
- 关联：constitution.md §2.1、§2.2；design.md §2.4、§5；README §四
- 作者：架构评审（EA）

## 背景

业务需区分「严重问题直接退回」与「轻微风险仍可审批但提醒审批人」。若采用二进制通过/失败：
费用超限（如餐饮 1200 > 限额 1000）这类本可由审批人裁量的轻微风险会被直接拒绝，造成误伤、
拉长报销周期；而重复报销、金额异常等严重问题必须硬性拦截。此外 AI 调用失败（系统异常）
不应等同于业务「失败」，需独立处理。

## 方案

- **A. 二态（通过 / 失败）**：实现简单，但无法表达「可审批但需注意」的中间态。
- **B. 三态 + 错误态**：通过(PASS) / 预警(WARNING) / 拦截(BLOCK)，外加系统错误(ERROR)。
  - 通过：无异常，自动进入审批；
  - 预警：轻微风险（超限、即将过期），仍进审批并标注；
  - 拦截：严重异常（重复 / 金额异常 / 字段缺失），直接退回；
  - 错误：AI 系统异常，转人工录入兜底。

## 决策

选择 **方案 B**。理由：

1. 精准映射业务语义，消除需求/设计/代码间术语不一致（design.md §2.4）；
2. 预警态保留人类审批裁量权，呼应宪章 §2.2「AI 辅助而非替代」；
3. 错误态与业务拦截解耦，保障降级路径清晰。

## 影响

- 校验状态机统一为四值枚举（`CheckStatus`：PASS / WARNING / BLOCK / ERROR），前端状态标签据此渲染；
- 自动填单仅对 通过/预警 启动审批流，拦截退回用户，错误转人工；
- 规则引擎需为每项检测标注严重程度（严重=拦截 / 警告=预警 / 提示=通过但记录）；
- 注意：需求 R2.6「失败」对应本系统状态「拦截」（design.md §2.4 备注）。
