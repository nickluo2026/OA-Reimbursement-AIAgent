# ADR-008：Agent 注册中心设计（插件化扩展多 AI Agent）

- 状态：已采纳
- 日期：2026-07（V1.4 已实现）
- 关联：constitution.md §2.4；design.md §8.5、§16.5、§16.6；ADR-007
- 作者：架构评审（EA）

## 背景

多票据类型（发票/行程单/火车票/机票）需各自独立的 Agent，若硬编码在 Graph 中则新增类型仍需改编排代码；
需统一的注册/发现机制实现插件化扩展，落实宪章 §2.4「新增票据类型仅需注册新 Agent 节点」原则。

## 方案

- **A. 硬编码 Agent 列表**：新增类型必须修改编排核心代码，违反开闭原则。
- **B. Agent 注册中心（装饰器注册 + 配置驱动路由）**：每个 Agent 通过 `@register_agent` 注册元信息，
  编排层按票据类型条件边动态选择节点，新增类型零侵入。

## 决策

选择 **方案 B**，设计 **Agent 注册中心**（`skill/orchestrator/registry.py`）：

1. 每个 Agent 通过 `@register_agent` 装饰器注册元信息（名称、支持的票据类型、入参/出参 Schema）；
2. 编排层根据票据类型路由（条件边）动态选择 Agent 节点；
3. 新增票据类型只需新增 Agent 文件并注册，无需修改编排核心代码（满足开闭原则）。

## 影响

- Agent 定义从编排逻辑中解耦，独立存放于 `skill/agents/`；
- 提供统一 Agent 抽象基类（`base_agent.py`），规范输入/输出契约；
- 支持运行时动态加载，为 V2.0 多票据并行奠定基础；
- 风险与缓解：
  - Agent 间数据契约不一致 → 强制继承基类 + Schema 校验；
  - 注册顺序依赖 → 注册中心去重，启动时校验完整性。
