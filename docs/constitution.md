# 企业报销智能化系统 — 项目宪章

> 本文档定义项目的核心原则、治理规则与开发规范。所有设计决策、代码提交与流程变更须遵守本宪章。

---

## 1. 核心愿景

打造端到端的**智能报销自动化系统**：用户上传票据 → AI 自动识别、校验 → OA 自动填单流转 → 审批归档。让报销从「填表」变为「确认」，将人工处理时间降低 80% 以上。

---

## 2. 核心原则

### 2.1 准确优先（Accuracy First）

- **金额校验零容忍**：任何涉及金额的计算与校验不得出现偏差，金额不匹配的票据必须拦截，不允许「放水」。
- **关键字段提取率 ≥ 95%**：发票号码、日期、金额、税号等核心字段提取准确率须持续达标。
- **校验结果可解释**：每次校验须给出明确的通过/失败理由与差异说明，不可仅返回布尔值。

### 2.2 AI 辅助而非替代（AI-Assisted, Not AI-Replaced）

- AI 负责**识别、校验、预警**，最终审批决策权始终在**人类审批人**手中。
- AI 识别结果必须以**可视化对比**形式呈现（原始票据 vs 提取结果），确保人工可复核。
- AI 校验失败的场景，系统必须保留**人工录入兜底**路径。

### 2.3 安全合规（Security & Compliance）

- **数据加密**：票据图片、发票信息等敏感数据在传输（全链路 HTTPS）与存储（**AES-256 加密**）环节均须加密。
- **最小权限**：API Key、数据库凭证等通过环境变量或密钥管理服务管理，不得硬编码。
- **敏感信息脱敏**：手机号、银行账号等敏感字段在展示与日志中脱敏（如 `138****1234`），非必要不采集。
- **审计可追溯**：每次 AI 调用、数据修改、审批流转须记录完整操作日志；**审计日志不可删除**，支持事后追溯。
- **数据保留与生命周期**：报销凭证保留 **≥ 7 年**（符合税务合规）；数据生命周期为 产生 → 活跃使用 → 归档（7 年）→ 安全销毁。
- **隐私保护**：用户个人信息遵循数据最小化原则，非必要不采集。

### 2.4 模块解耦（Modularity & Loose Coupling）

- **识别层、校验层、流程层分离**：OCR/AI 识别、规则校验、OA 流程流转作为独立模块开发，通过 API 通信。
- **编排与业务解耦**：采用开源智能体编排平台 **LangGraph**（StateGraph 图式工作流）管理工作流，编排逻辑与业务 Agent 解耦。新增票据类型仅需注册新 Agent 节点并扩展条件边路由，不修改编排核心代码（见设计文档 §16、ADR-007/ADR-008）。
- **票据类型可扩展**：新增票据类型（火车票、机票、餐饮发票等）时，只需新增对应 Schema 和 Prompt，不修改核心逻辑。
- **OA 系统可替换**：与 OA 的集成通过适配器模式封装（设计文档 §14），切换其他 OA 系统时变更范围最小化。

### 2.5 快速反馈（Fast Feedback Loop）

- 单张票据的**识别 + 校验响应时间 ≤ 10 秒**。
- 识别失败或异常时，**立即**向用户反馈明确错误信息，不允许静默失败。
- 流程状态变更须实时通知相关方（邮件 / 企业微信 / 钉钉）——**规划中**；当前版本由审批 / 财务工作台主动拉取状态，未实现主动推送。

### 2.6 多角色协作（Multi-Role Collaboration）

- 系统支持 **5 类角色独立工作台**：员工、主管、财务、出纳、系统管理员，各角色仅可见权限范围内的功能（对齐 README §二 与 `prototype.html`）。
- **角色权限矩阵须可视化**：前端提供角色说明面板，展示每个角色的主要操作与使用场景，帮助用户理解自身权限边界。
- **主管**拥有最终审批决策权，AI 仅提供校验结果与预警标识，不替代人类决策（呼应 §2.2）；审批人可「通过 / 驳回 / 转审」。
- **财务职责分离（Separation of Duties）**：财务（岗 FIN-001，仅归档）与出纳（岗 FIN-002，仅打款）为两个独立角色，系统强制「打款人 ≠ 归档人」，杜绝一人包办复核与发放（对齐 README §二 与代码 `web/app.py`）。
- **大额会签**：报销金额 ≥ 1 万元时，在对应审批级别基础上增加一位审批人（两人会签），会签开关由系统管理员配置（默认启用）。
- **系统管理员**负责维护费用限额 / 异常规则 / 审批权限，配置变更须全部记入审计日志；并可通过用量统计面板查看 AI 调用消耗。

## 3. 开发规范

### 3.1 代码规范

| 规范项 | 要求 |
|--------|------|
| 语言 | Python 3.10+ |
| 代码风格 | 遵循 PEP 8，使用 `black` 格式化 |
| 类型注解 | 所有公开函数必须添加类型注解 |
| 文档字符串 | 模块、类、公开方法须有 docstring |
| 依赖管理 | 运行时依赖锁定于 `requirements.txt`；开发 / 构建配置（black/ruff/pytest 等）置于 `pyproject.toml` 的 dev extras |
| 数据库 | SQLAlchemy ORM；当前 SQLite（V1.4 验证），目标 MySQL 8.0（V2.0 生产，见 ADR-005） |
| API Key 管理 | 仅通过 `.env` 文件加载，`.env` 加入 `.gitignore` |
| 前端规范 | 前端采用原生 HTML/CSS/JS（Flask 模板 + 静态资源），遵循语义化标签；5 角色工作台按角色路由与权限隔离；样式使用 CSS 变量统一设计令牌 |

### 3.2 AI 调用规范

```python
# ✓ 正确：通过 Function Call 获取结构化数据
payload = {
    "model": "deepseek-v4-flash",
    "tools": INVOICE_TOOLS,
    "tool_choice": "auto",
    "temperature": 0.0  # 提取任务固定 0，保证确定性
}

# ✓ 正确：校验结果必须包含明确说明
{
    "is_valid": False,
    "validation_result": "发票金额1200 > 申请金额900，超出300，不通过"
}

# ✗ 错误：仅返回布尔值，无解释
{"is_valid": False}
```

- **Temperature 固定为 0.0**：所有提取类任务使用确定性输出。
- **Function Call 优先**：结构化数据提取使用 `tools` 机制，避免正则解析自然语言。
- **异常兜底**：AI 调用超时或失败，必须有明确的错误码和用户提示。

### 3.3 Git 提交规范

```
<type>: <简短描述>

[详细说明（可选）]

类型：feat / fix / refactor / docs / test / chore
示例：
feat: 新增行程单识别模块
fix: 修复金额校验浮点精度问题
docs: 更新 API 接口文档
```

### 3.4 测试规范

- **单元测试**覆盖所有核心函数（PDF 提取、Schema 校验、规则引擎）
- **集成测试**覆盖完整识别→校验→流程链路
- AI 相关测试使用固定 mock 数据，避免依赖外部 API
- 上线前必须通过全量回归测试

## 4. 决策机制

### 4.1 架构决策记录（ADR）

重大技术决策须以 ADR 形式记录，包含：
- **背景**：为什么需要做这个决策
- **方案**：有哪些可选方案
- **决策**：选择哪个方案
- **影响**：这个选择的 trade-off 是什么

> 已采纳的 ADR 索引见设计文档 §10，包括：
> - ADR-001：选择 DeepSeek 作为 AI 引擎
> - ADR-002：使用 Function Call 结构化输出
> - ADR-003：三态校验模型（通过/预警/拦截）
> - ADR-004：YAML 配置驱动规则引擎
> - ADR-005：数据库选型策略（SQLite → MySQL）
> - ADR-006：票据文件存储策略（本地 → OSS）
> - ADR-007：引入 LangGraph 智能体编排平台
> - ADR-008：Agent 注册中心设计
>
> 完整 ADR 存放于 `docs/adr/` 目录。

### 4.2 审批权限

| 变更类型 | 谁来决定 |
|----------|----------|
| 核心架构变更 | 技术负责人审批 |
| AI 校验规则调整 | 技术 + 业务双审批 |
| 新增票据类型 | 产品经理 + 技术负责人 |
| 金额阈值调整 | 财务部门审批 |

## 5. 质量门禁（Definition of Done）

一个功能「完成」的判定标准：

- [ ] 代码通过 Code Review
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 集成测试通过
- [ ] 错误场景有明确的错误处理与用户提示
- [ ] API 文档已更新
- [ ] 相关 ADR 已记录（如有架构变更）

## 6. 风险与约束

| 风险 | 缓解措施 |
|------|----------|
| AI 识别准确率不达标 | 持续标注反馈数据，建立 Bad Case 库微调模型 |
| 外部 API 不可用 | 核心链路有降级方案，支持人工录入兜底 |
| 票据格式多样 | Schema 设计预留扩展字段，Prompt 持续迭代 |
| 数据安全合规 | 敏感数据脱敏，定期安全审计 |
|   OA 版本升级 | 适配器层隔离，版本兼容测试 |
