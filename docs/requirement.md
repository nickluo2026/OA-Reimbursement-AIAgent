# 企业报销智能化系统 — 需求文档

## 1. 项目背景

企业日常报销流程依赖人工录入发票、行程单信息，效率低、易出错。本项目基于开源智能体编排平台 LangGraph 与 DeepSeek 大模型，与 OA 系统集成，实现报销单据的智能识别、自动校验与流程流转，提升报销处理效率与合规性。

## 2. 目标用户

系统支持 5 类角色，每类角色拥有独立工作台与操作权限。角色权限矩阵如下（与原型 `prototype.html` 角色说明面板对齐）：

| 角色 | 主要操作 | 使用场景 |
|------|---------|---------|
| **👤 普通员工** | 上传票据 → 填写事由与金额 → 提交校验 → 等待审批结果 | 提交日常差旅、餐饮、住宿等报销 |
| **👔 审批领导** | 查看待审报销单 → 查看 AI 校验结果与预警 → 通过 / 驳回 / 转审 | 审核下属报销申请 |
| **💼 财务复核** | 复核 AI 校验结果 → 确认归档 | 财务复核与归档（岗 FIN-001，仅复核/归档，不可打款） |
| **🏦 出纳打款** | 发起费用发放 → 银行回单归档 | 打款与回单归档（岗 FIN-002，须与归档人不同人，防舞弊） |
| **⚙️ 系统管理员** | 配置费用限额 / 异常规则 / 审批权限 → 查看审计日志 | 维护报销制度规则 |

> **职责分离（四眼原则）**：「确认归档」与「发起费用发放」为互斥操作，须由不同财务人员完成。系统强制校验「打款人 ≠ 归档人」，同一人操作将被拦截，防止"自己确认、自己付款"的舞弊风险。

> 各角色登录后进入对应工作台，仅可见权限范围内的功能 Tab（见设计文档 §17 前端架构设计）。

## 3. 功能需求

### 3.1 阶段一：申请与预处理

| 编号 | 需求 | 优先级 |
|------|------|--------|
| R1.1 | 用户可扫描或上传发票、行程单等报销凭证图片/PDF | P0 ✅ 已实现 |
| R1.2 | 用户填写基本报销详情（申请金额、事由、申请日期等） | P0 ✅ 已实现 |
| R1.3 | 系统接收图片与单据，自动形成并启动报销工作流程 | P0 ✅ 已实现 |
| R1.4 | 系统支持多文件并发上传与批量校验（前端拖拽多选，后端并发处理） | P1 ✅ 已实现 |

### 3.2 阶段二：AI 智能读取与校验

> **校验执行顺序**：OCR 提取 → 异常检测（前置拦截）→ 分类限额校验。异常检测若拦截则直接返回，跳过后续校验。

| 编号 | 需求 | 优先级 |
|------|------|--------|
| R2.1 | 系统通过 API 调用 DeepSeek AI Agent 进行图像识别与 OCR，提取发票代码、日期、金额等关键字段 | P0 ✅ 已实现 |
| R2.2 | 系统通过 NLP 与数据语义分析，识别费用类型（差旅/餐饮/住宿/交通/办公/其他） | P0 ✅ 已实现 |
| R2.3 | 系统连通外部 API（增值税发票查验平台）进行真实性核验 | P1  ⚠️ 框架已实现，真实平台未接|
| R2.4 | 系统根据公司制度规则集进行核对（金额限额、费用分类、审批权限） |  P0 ✅ 已实现|
| R2.5 | 系统进行异常检测（前置拦截），识别字段缺失、格式错误、重复报销、金额异常（含申请金额不足）、票据过期、票据即将过期（剩余有效期<30天预警）等风险 | P0 ⚠️ 大部分已实现，重复报销未接入  |
| R2.6 | AI Agent 返回校验结果：通过 / 预警 / 拦截 / 错误（AI 调用失败等系统异常，转人工兜底） | P0 ✅ 已实现  |
| R2.7 | 系统支持小额免审：发票金额 ≤ 100 元跳过分类限额校验 | P1 ✅ 已实现 |
| R2.8 | 系统支持行程单 OCR 提取与合理性校验（行程数、金额、时间等） | P1 ✅ 已实现 |

### 3.3 阶段三：流转与审批

| 编号 | 需求 | 优先级 |
|------|------|--------|
| R3.1 | 系统自动填充报销单字段（日期、金额、摘要、费用分类、发票号码、销售方名称、税额等） | P0 ✅ 已实现 |
| R3.2 | 系统显示 AI 校验意见与预警标识（费用超限、重复报销、票据过期、金额异常、字段缺失） | P0 ✅ 已实现 |
| R3.3 | 报销单按金额阶梯审批流转（≤3000元直属领导→财务；≤10000元部门总监→财务；≤50000元VP→财务；>50000元CEO→财务），金额≥10000元需两人会签，支持驳回与转审 | ✅ 已实现|
| R3.5 | 前端展示执行流水线可视化（LangGraph 节点逐步执行动画：路由→OCR→异常检测→分类/校验，含节点名称、调用工具及执行详情） | P1 ✅ 已实现 |
| R3.6 | 系统支持 5 角色独立工作台：普通员工（报销校验）、审批领导（待审工作台）、财务复核（财务复核工作台）、出纳打款（出纳打款工作台）、系统管理员（系统配置+审计日志+用量统计），按角色路由可见 Tab | P0 ✅ 已实现  |
| R3.8 | 审批领导工作台展示待审报销单列表，含 AI 校验结果标签（通过/预警/拦截）、风险等级，支持通过/驳回/转审/查看明细操作 | P0 ✅ 已实现 |
| R3.9 | 财务复核工作台展示待归档报销单，支持确认归档（记录归档人）；出纳打款工作台展示已归档待打款报销单，支持发起打款（强制校验打款人≠归档人）、银行回单回写归档，含待复核/待归档/待打款/已打款统计 | P0 ✅ 已实现 |

### 3.4 阶段四：用量统计与审计

| 编号 | 需求 | 优先级 |
|------|------|--------|
| R4.1 | 系统管理员可查看 DeepSeek API 用量统计：总调用次数、Token 总量（输入/输出）、预估费用（CNY）、平均延迟、成功率 | P1 ✅ 已实现 |
| R4.2 | 用量统计支持每日调用趋势柱状图（近 7 天，柱高=Token 消耗，柱顶=调用次数），鼠标悬停查看明细 | P1 ✅ 已实现 |
| R4.3 | 用量统计支持按调用类型（发票OCR/行程单OCR/异常检测/分类限额/Vision API）分类统计 Token 占比与费用 | P1 ✅ 已实现 |
| R4.4 | 用量明细记录支持按日期 / 调用类型 / 状态筛选，展示 Request ID、模型、Token 数、延迟、费用 | P1 ✅ 已实现 |
| R4.5 | 系统管理员可查看审计日志：记录所有用户操作（提交/审批/归档/打款/回单归档）、AI 校验、系统配置变更，含操作人、角色、操作类型、对象、结果、IP | P0 ✅ 已实现 |
| R4.6 | 系统管理员可配置费用限额（交通/住宿/餐饮/行程单阈值）、异常检测规则开关、审批权限分级 | P0 ✅ 已实现 |

## 4. 非功能需求

| 编号 | 需求 | 说明 |
|------|------|------|
| N1 | 性能 | 前端展示异步进度动画，用户不被空白页阻塞 | ✅ 已实现
| N2 | 性能 | 单张票据端到端 AI 识别与校验总耗时 ≤ 10 秒 | ✅ 完成自动化验证
| N3 | 安全性 | 审计日志金额脱敏| ✅ 已实现
| N4 | 扩展性 | 支持后续新增票据类型（火车票、机票等），插件化 Schema + 配置驱动规则 | ✅ 已实现

## 5. 集成需求

- **OA 系统**：流程启动、表单填充、审批流转、数据归档（见设计文档 §1.3、§14 OA 适配器层）❌ 未实现
- **LangGraph**：智能体编排平台，StateGraph 工作流编排（票据路由 → 发票/行程单双 Agent 分支，见设计文档 §16）✅ 已实现
- **DeepSeek AI Agent**：OCR 识别、语义分析、智能校验 ✅ 已实现
- **增值税发票查验平台**：发票真实性核验 ⚠️ Mock
- **数据库**：当前 SQLite（功能验证），目标 MySQL 8.0（生产环境，见设计文档 §1.3、ADR-005）⚠️ 部分

> 重大技术决策记录（ADR-001 至 ADR-008）见设计文档 §10。
